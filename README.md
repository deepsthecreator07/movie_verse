# MovieVerse

A Flutter application where multiple users can each save movies they want to watch, and the app surfaces movies that everyone has saved — making group movie night decisions easy.

---

## What the App Does

MovieVerse has four main screens that flow from one into the next:

| # | Screen | Purpose |
|---|--------|---------|
| 1 | **Users** | Lists all users (fetched from Reqres API + any locally created ones). Tap a user to see their movies. |
| 2 | **Add User** | Create a new user with a name and movie taste description. Works offline — the user is stored locally and synced to the API when connectivity returns. |
| 3 | **Movies** | Browse movies (from TMDB) and bookmark the ones you want to watch. Each card shows a live count of how many users have saved that movie. |
| 4 | **Movie Detail** | Full-screen poster with rating, overview, and a list of which users have already saved that movie. |
| 5 | **Saved Movies** | A user's personal watchlist — all the movies they've bookmarked. |
| 6 | **Matches** | Movies that **two or more** users have saved. This is the group recommendation engine — if everyone has bookmarked the same film, it appears here. |

**Offline behaviour:** The app caches both users and movies into a local SQLite database on first load. If the device loses internet, all browsing and saved-movie lookup continues from the cache. Newly created users are queued with a `pendingSync` flag and uploaded in the background once connectivity is restored.

## Design System

### Color Palette — "Cinematic Dark"

The entire UI uses a custom dark theme defined in `app_colors.dart`. Every color is hand-picked; no Material defaults are used for primary surfaces.

| Role | Color | Hex |
|------|-------|-----|
| Primary | Vivid Indigo | `#6C63FF` |
| Accent | Warm Amber/Gold | `#FFB74D` |
| Background | Near-black Deep Blue | `#0D0D1A` |
| Card Surface | Dark Navy | `#1A1A2E` |
| Elevated Surface | Slightly Lighter Navy | `#2D2D4A` |
| Text Primary | Off-white | `#F5F5F5` |
| Text Secondary | Muted Lavender | `#B0B0C8` |

### Typography

`google_fonts` is used to load **Poppins** (headings) and **Inter** (body text) at runtime. Both are declared in `app_text_styles.dart` and referenced as named constants throughout the UI — no inline `TextStyle(...)` calls in widget files.

### Component Patterns

- **Shimmer skeletons** replace spinners during initial data loads.
- **Hero animations** link the movie poster thumbnail on the grid to the full-screen detail view.
- **Staggered list animations** on the Users page use a per-index delay so tiles cascade in rather than all appearing at once.
- **Floating SnackBars** are used for all user feedback (movie saved, user added, sync complete) — no blocking dialogs.

### The User → Saved Movies Relationship

`saved_movies` is a standard many-to-many join table. One user can save many movies; one movie can be saved by many users.

```
users ──< saved_movies >── movies
   id         user_id          id
              movie_id
```

This design powers three distinct read paths, all in `app_database.dart`:

1. **Personal watchlist** (`watchSavedMoviesForUser`): `INNER JOIN movies ON movies.id = saved_movies.movie_id WHERE user_id = ?` — returns the movies for one user's watchlist page.

2. **Save count per movie** (`watchSaveCountForMovie`): `COUNT(id) WHERE movie_id = ?` — the live badge on each movie card showing how many users want to watch it.

3. **Matches** (`watchMatches`): A raw SQL query that counts saves per movie and filters to those saved by **two or more** users — `HAVING COUNT(user_id) >= 2`. This is the core feature of the app.

All three use Drift's `.watch()` which returns a `Stream`, so the UI reacts in real-time when any user saves or unsaves a movie without needing a manual refresh.

## Non-Obvious Decisions

### 1. Two separate user ID spaces

The `users` table has its own autoincrement `id` (local) and a nullable `remote_id` (from Reqres). All foreign keys in `saved_movies` point to the **local** `id`. This means a user can save movies immediately after being created offline — before they have a `remote_id` — and the relationship is never broken when the sync eventually fills in `remote_id`.

### 2. API keys are never in source control

`ApiConstants` reads keys with `String.fromEnvironment(...)`, which bakes them in at compile time via the `--dart-define-from-file` flag. They are stored in a local `.env` file that is listed in `.gitignore`. To run the app:

```bash
flutter run --dart-define-from-file=.env
```

The `.env` file structure (create this locally):
```
TMDB_API_KEY=your_key_here
OMDB_API_KEY=your_key_here
REQRES_API_KEY=your_key_here
```

VS Code users: a `.vscode/launch.json` is included that passes this flag automatically when pressing **Run**.

### 3. Matches are computed locally, not from an API

There is no server-side "matches" endpoint. The Matches page runs a SQLite `HAVING COUNT >= 2` query over the local `saved_movies` table. This keeps the feature fully offline and avoids any additional API dependency. The tradeoff is that matches only reflect movies saved by users on **this device** — it is not a cross-device social feature.

### 4. Optimistic UI for saving movies

When a user taps the bookmark on a movie, the BLoC immediately updates the saved set in state (optimistic update) before the database write completes. If the write fails, the original state is restored. This removes all perceptible latency from the save action.

### 5. Background sync uses WorkManager, foreground sync uses a direct call

When the app creates a user offline, it schedules a WorkManager one-off task constrained to `NetworkType.connected`. WorkManager will fire this even if the app is killed. Additionally, when the app is **in the foreground** and connectivity is restored, `ConnectivityService`'s stream triggers an immediate foreground sync via `UserRepository.syncPendingUsers()`, which is faster and shows the user a SnackBar on completion. Both paths converge on the same `syncPendingUsers()` implementation.

### 6. `Movies.id` is not autoincrement

TMDB movie IDs are used directly as the primary key. This means the same movie fetched across different pages or at different times always maps to the same row — `insertOnConflictUpdate` acts as an upsert. If autoincrement were used, repeated fetches of the same movie would create duplicate rows.
